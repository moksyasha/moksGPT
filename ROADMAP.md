# moksGPT — roadmap

цель: собрать LLM с нуля руками, понять каждый блок (attention, KV-кэш, RoPE,
GQA, RMSNorm, SwiGLU), а потом дорастить до архитектуры Qwen3 — той самой, что
стоит backbone'ом внутри нашей TTS OmniVoice. дальше — мостик в TTS.

принцип: **никакого копипаста.** каждый блок пишу сам, claude объясняет и
проверяет понимание, сверяемся с реальным конфигом Qwen3-0.6B (см. ниже).
после каждого этапа — рабочий чекпоинт, который я могу запустить и потрогать.

## этап 2 — self-attention руками (2-3 дня) ← ядро всего

здесь происходит «клик». пишем внимание как матмул, без `nn.MultiheadAttention`.

- [ ] одна голова: Q,K,V проекции, `scores = Q@K.T / sqrt(d)`, softmax, `@V`
- [ ] causal mask (нижнетреугольная) — почему нельзя смотреть в будущее
- [ ] multi-head: split на головы, параллельно, concat, выходная проекция
- [ ] собрать блок: attention + MLP + residual + LayerNorm (пока обычный)
- [ ] стопка блоков → мини-GPT, обучить на shakespeare

выход: генерит похожий на шекспира текст. я могу объяснить вслух каждую строку
attention. **это и есть «понял аттеншены».**

контрольные вопросы к себе (claude спросит):
- зачем делить на `sqrt(d_k)`?
- что в KV меняется между causal mask и его отсутствием?
- почему residual важен для глубины?

## этап 3 — KV-кэш и инференс (1 день)

мостик к моему основному скиллу (инференс/mlops).

- [ ] добавить KV-cache в attention для генерации
- [ ] замерить скорость генерации с кэшем и без — понять выигрыш руками
- [ ] связать с тем что я уже знаю: почему GQA (этап 5) режет размер кэша

выход: бенч «токенов/сек с кэшем vs без», я объясняю откуда ускорение.

## этап 4 — нормальная токенизация (0.5 дня)

- [ ] заменить char-level на BPE (`tiktoken` или обучить свой мини-BPE)
- [ ] понять: subword, почему vocab Qwen3 = 151676, спец-токены

выход: модель на BPE, меньше контекста на то же кол-во текста.

## этап 5 — апгрейд до Qwen3 (1 неделя) ← цель маршрута

добавляем по ОДНОМУ компоненту, после каждого сверяемся с `llm_config` нашей
OmniVoice и проверяем что loss не сломался:

- [ ] `LayerNorm → RMSNorm` (eps 1e-6) — те самые что фьюзим в Triton
- [ ] learned pos-emb `→ RoPE` (rope_theta = 1e6, head_dim 128)
- [ ] MHA `→ GQA` (16 attention heads, 8 KV heads) — половина KV-кэша
- [ ] MLP `→ SwiGLU` (silu, intermediate 3072)
- [ ] tie_word_embeddings, конфиг 28 слоёв / hidden 1024

выход: мини-Qwen3, который я понимаю до тензора. архитектурно = backbone
OmniVoice, только обучен на тексте а не на аудио-кодах.

## этап 6 — мостик в TTS (по желанию, отдельный спринт)

теперь LLM-часть OmniVoice не магия. разбираем надстройку по слоям:

- [ ] **neural audio codec** (EnCodec/DAC/Higgs) отдельно: как wav → дискретные
      коды и обратно. без этого TTS не понять.
- [ ] **MaskGIT/SoundStorm** декодинг на игрушке: параллельный unmask, N шагов
- [ ] собрать text → audio_heads (8×1025) → codes → codec → wav как у OmniVoice
- [ ] вернуться к `analysis.md` в api_core — он весь станет читаемым

**torch** (всплывает по ходу, дрилл при первой встрече):

- [ ] индексация и срезы тензоров, `torch.stack`, `torch.randint` — этап 0

- [ ] matmul `@` / `torch.matmul`, `.transpose`/`.T` — этап 2
- [ ] `softmax`, `masked_fill`, `tril` — этап 2
- [ ] `nn.Module`, `nn.Linear`, `nn.Embedding`, параметры и `forward` — этап 1-2

**math** (интуиция + минимум формул, дрилл при встрече):

- [x] производная/градиент = «куда крутить веса» (интуиция, без вывода) — этап 1

правило: фундамент тоже идёт в мини-экзамены. claude спрашивает не только «что
делает код», но и «что делает эта torch-операция и почему именно она».

## правила работы (claude обязан соблюдать)

это контракт. claude следит за их соблюдением, я прошу — он не «жалеет».

1. **I do → we do → you do** на каждый блок:
   - *I do* — claude даёт мини-лекцию + рабочий пример с комментами, я читаю/понимаю;
   - *we do* — claude просит поменять мелочь (число голов, убрать mask), я правлю, обсуждаем;
   - *you do* — claude даёт каркас с TODO-дыркой, я заполняю сам. дырки растут по
     мере уверенности: этап 0-1 claude пишет ~90%, к этапу 5 пишу я.
2. **статьи обязательны.** перед нужным блоком claude ЗАСТАВЛЯЕТ прочитать
   секцию (по треку статей выше), НЕ даёт код пока не прочитал, потом
   спрашивает по прочитанному. не сачковать.
3. **мини-экзамены постоянно.** после каждого блока (не только этапа) claude
   гоняет меня по контрольным вопросам — отвечаю вслух/текстом своими словами.
   объяснил чисто → идём дальше. поплыл → возвращаемся, разбираем заново.
   claude не пропускает экзамен «чтобы не тормозить».
4. **не перепрыгиваю этапы.** каждый этап = рабочий чекпоинт в git, коммитим
   перед следующим.
5. **модели маленькие.** ресурсы есть (4090), но цель понять, а не натренить
   sota. учимся быстро итерироваться, не ждать часами.
6. claude **не даёт готовое решение** на *you do*-дырку, даже если прошу —
   max подсказка/наводящий вопрос. готовый код только на *I do*-примерах.
